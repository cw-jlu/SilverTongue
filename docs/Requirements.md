# SilverTongue (灵舌) - 开发要求与作品规范

本文档整理了 SilverTongue 项目的核心开发要求、评审规则以及作品提交规范，作为项目开发全周期的质量把控和提交基准。

---

## 1. 核心任务要求

本项目致力于开发一款 **AI 英语口语练习工具**，模拟真实对话训练场景。

### 1.1 必备功能模块
*   **场景选择**：支持面试、点餐、会议等多种预设场景，提供丰富的语境选择。
*   **实时语音对话**：实现端到端的高效、流畅语音交互，还原真实的交流体验。
*   **发音评测**：对用户的口语发音进行多维度（准确度、流利度、完整度）的专业评估。
*   **纠错与反馈**：包括实时语法纠错、地道表达优化建议（Native Rewrite）以及课后总结报告。

### 1.2 关键性能指标
*   **低延迟交互**：AI 响应的首次出字延迟（TTFT）需控制在 **1.5秒** 以内，确保对话的自然与流畅。
*   **纠错精准度**：提供音素级的发音诊断，精准定位发音错误，且反馈时机合理。
*   **学习成效可量化**：通过数据热力图、进度追踪与智能闪卡（SRS），为用户提供清晰、可量化的学习成效反馈。

---

## 2. 评审规则

作品的最终评审将严格按照以下三个维度进行评分：

1.  **作品完整度与创新性 (40%)**
    *   各功能模块实现的合理性与完整性。
    *   人机交互的流畅度与用户体验。
    *   产品的创新性设计与亮点功能。
2.  **开发过程与质量 (40%)**
    *   系统架构的清晰度、代码的可读性与逻辑的健壮性。
    *   **PR (Pull Request) 提交规范**：PR 需分布均匀，严禁在最后时刻进行突击提交。
    *   **Commit 提交历史**：Commit 记录应体现全周期的持续交付与分步开发过程。
3.  **演示与表达 (20%)**
    *   Demo 视频的讲解清晰度、逻辑条理性与产品功能展示度。

---

## 3. 作品提交规范 (Submission Rules)

### 3.1 仓库管理规范
*   代码必须提交至公开的 **GitHub/Gitee** 仓库。
*   仓库内必须包含：`完整源代码` + `README.md 文档` + `Demo 视频链接`。
*   项目需采用多模块分目录管理（如前端放置于 `/frontend`，后端放置于 `/backend`）。

### 3.2 持续交付与 PR 规范
*   **严禁一次性导入代码**：从项目启动起，必须有持续、有规律的 Commit 和 PR 记录。
*   **PR 粒度控制**：每个 PR 应只实现**单一功能**，保持细粒度提交，便于 Code Review。
*   **PR 描述四要素**：
    1.  **标题**：一句话清晰说明新增或修改的核心内容。
    2.  **功能描述**：说明该功能的作用、解决的痛点及用户使用方式。
    3.  **实现思路**：简述技术选型、核心算法或核心业务逻辑。
    4.  **测试方式**：说明如何验证该功能已正常运行（包含测试用例或截图）。

### 3.3 Demo 视频规范
*   视频需有完整的声音讲解，覆盖项目的所有核心模块和亮点功能演示。
*   需上传至外部主流平台（如 B站、百度网盘等），并将链接清晰地放入 `README.md` 的醒目位置。

---

## 4. 诚信与合规

*   **自主研发**：所有核心代码必须自主完成。若代码重复率超过 50% 或被确认存在抄袭行为，将直接取消参赛资格。
*   **第三方开源引用**：项目中引用的所有第三方库、框架或开源组件，必须在 `README.md` 中详细列明。
*   **知识产权**：作品的知识产权归提交者所有，但主办方拥有免费演示与宣传的权利。

---

## 5. SilverTongue 开发准则

为了保证代码库的整洁和开发过程的透明，开发团队需严格遵循以下准则：
1.  **分步小额提交**：每完成一个数据库表结构设计、一个 API 接口定义、或一个前端组件实现，都必须进行一次 Commit。
2.  **文档先行原则**：任何功能变更或表结构修改，必须首先在 `PRD.md` 和 `Database_Design.md` 中进行同步更新，确保文档与代码保持一致。
